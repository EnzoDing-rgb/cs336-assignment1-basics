# Batch size experiment（TinyStories）

**问题：** 在同样多的训练数据（同 token / 同量级 FLOPs）下，batch size 变大是更好、更差，还是只影响速度？

**交付图（左右一图，含 8→512）：**  
[`artifacts/plots/batch_size/batch_size_summary.png`](../artifacts/plots/batch_size/batch_size_summary.png)

---

## 1. 设置（单一变量）

| 固定 | 取值 |
|------|------|
| 数据 / tokenizer / 模型 | 与 `tinystories_small.yaml` 相同 |
| 总 token 预算 | \(20000\times 32\times 256 = 1.6384\times 10^8\) |
| 步数 | \(\mathrm{iters}(B)=\mathrm{budget}/(B\cdot 256)\) |
| LR 日程形状 | warmup / cosine 随 `iters(B)` 比例缩放 |
| 每档初始化 | 从零训练（不复用旧权重） |

| 变 | 做法 |
|----|------|
| `batch_size` | `8, 16, 32, 64, 128, 256, 512` |
| `lr_max` | 以 \(1.8\times 10^{-3}\times(B/32)\) 为中心，短跑 20% token 试 `{0.5×,1×,2×}`，长跑只用赢家 |

日志按 token 对齐（与基准「每 \(32\times 256\times 600\) token」同间隔）。

---

## 2. 结果总表

| \(B\) | 选中 `lr_max` | final valid | 墙钟 (min) | steps |
|------:|-------------:|------------:|-----------:|------:|
| 8 | 0.0009 | 1.537 | 42.4 | 79200 |
| 16 | 0.0018 | 1.447 | 35.4 | 39600 |
| 32 | 0.0018 | 1.425 | 33.2 | 19800 |
| 64 | 0.0018 | 1.429 | 32.4 | 9900 |
| **128** | **0.0036** | **1.411** | **31.9** | 4950 |
| 256 | 0.0144 | 1.634 | 33.2 | 2475 |
| 512 | 0.0144 | 1.889 | 34.7 | 1216 |

---

## 3. 发现

1. **不是越大越好（质量）。**  
   同 token 下 valid 先降后升：甜区约 **32–128**，最佳 **128（1.411）**；**256→512 明显变差**（1.63→1.89）。  
   梯度更稳 ≠ 学得更好：噪声过小、更新次数过少、LR 难以再线性拉高，都会伤效果。

2. **墙钟也不会无限下降。**  
   \(8\to 32\)：GPU 从吃不饱到吃饱，墙钟 42→33 min。  
   \(32\to 128\)：每步时间大致 \(\propto B\)，总 FLOPs 相近 → **墙钟走平（~32 min）**。  
   再大到 256/512：并不更快，甚至略慢。  
   这符合「算力饱和后，加大 batch 只是把同样多的计算揉进更少、更贵的 step」。

3. **和「永远用最大 batch」相反：**  
   本设定应选 **质量最好且接近最快墙钟** 的档（约 **128**），而不是顶满显存的 512。

---

## 4. 备注

- 网格按 one-go 协议；256/512 在显存腾出后纳入同一套图与表（覆盖同一 PNG）。
- 短跑选 LR 为离散三档，不保证全局最优，但足以支撑趋势结论。
- 上一问 learning-rate 扫参见 [`learning_rate.md`](learning_rate.md)；锚在 `B=32`、`lr_max=1.8e-3`。本实验里 `B=32` 的 final valid ≈ 1.425，与那次对照一致。

---

## 5. 一句话

**同样多数据时，batch 有甜区：太小又慢又差一点，太大（256/512）明显伤 valid；墙钟在 GPU 吃饱后走平。本实验最佳约 `B=128`。**
